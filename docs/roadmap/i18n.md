# i18n — audit + roadmap

> **Statut** : audit posé. Provider maison en place, FR + EN
> déclarés ; **6 namespaces sur 13 sont vides** côté
> `packages/web/src/i18n/locales/` et l'app contient encore
> ~150 chaînes UI hardcodées en français + emails 100 % FR. Cette
> roadmap décrit comment passer de « bilingue sur le papier » à
> « bilingue pour de vrai ».
>
> **Mise à jour** : à chaque PR qui livre une étape, cocher la
> case correspondante. Si une décision technique change la
> stratégie (lib externe adoptée, provider remplacé, etc.), mettre
> à jour `documentation/Internationalisation.md` dans le **même
> commit** — pas en suivi.

Audit mené en parallèle sur `packages/web/src/` (UI) et
`packages/api/src/` (templates email). Les chiffres correspondent
à l'état au commit qui livre ce document. Référence
d'implémentation existante (à imiter) :
[`packages/web/src/i18n/locales/fr/auth.json`](../../packages/web/src/i18n/locales/fr/auth.json)
+ usages dans [`Login.tsx`](../../packages/web/src/app/pages/Login.tsx).

## État des lieux

### Infrastructure existante

[`packages/web/src/i18n/I18nProvider.jsx`](../../packages/web/src/i18n/I18nProvider.jsx)
expose un hook `useI18n()` :

- `t('namespace.key', { defaultValue, values })` — interpolation
  `{token}`, fallback automatique vers la langue par défaut (`fr`).
- `setLanguage(code)` — persiste dans `localStorage`
  (`nodea:language`) **et** dans `user_preferences` chiffré
  (sync cross-device une fois loggué·e).
- `availableLanguages` — alimenté par la const `SUPPORTED_LANGUAGES`
  du provider.

Détection initiale : `localStorage` → `navigator.language` → `fr`.

### Patterns problématiques

**Pluralisation par ternaire** — 12+ occurrences éclatées, aucune
helper centrale :

```tsx
// app/flow/Goals/index.tsx:56
total === 1 ? 'objectif' : 'objectifs'
// app/flow/Account/views/SecurityTab.tsx:12 (double s)
{count} passkey{count > 1 ? 's' : ''} enregistrée{count > 1 ? 's' : ''}
// app/flow/Goals/components/CarryOverDialog.tsx
affected.length === 1 ? 'va' : 'vont'  // verbe conjugué
```

Une migration vers `Intl.PluralRules` ou ICU MessageFormat est un
prérequis avant tout effort de traduction sérieux.

**Date hardcoding FR** — trois sources distinctes :

- [`core/i18n/date-fr.ts`](../../packages/web/src/core/i18n/date-fr.ts)
  contient `"Aujourd'hui"`, `"Hier"` en littéraux + utilise
  `Intl.DateTimeFormat('fr-FR', …)` partout.
- [`app/flow/Goals/lib/date-format.ts`](../../packages/web/src/app/flow/Goals/lib/date-format.ts)
  définit son propre tableau `FRENCH_MONTHS`.
- [`ui/dirk/ComposerModal.tsx`](../../packages/web/src/ui/dirk/ComposerModal.tsx)
  contient un troisième tableau de mois (Janvier–Décembre).

Aucune branche EN n'existe pour ces formats — un user en `en`
verra les mois et les labels de date rester en français.

**Codes machine traduits inline** — l'API renvoie des codes type
`'invalid_token'` / `'account_not_activated'` ; chaque page front
les traduit avec sa propre table en dur ([`Login.tsx:41-65`](../../packages/web/src/app/pages/Login.tsx#L41-L65)).
Pas de table partagée, donc une modif d'un wording impose de
chasser dans chaque page consommatrice.

**Emails 100 % FR** — ~48 chaînes hardcodées sur 8 templates dans
[`packages/api/src/services/email/templates/`](../../packages/api/src/services/email/templates/),
sans variant EN. Les emails déclenchés par le user (activation,
reset password, MFA bypass, invite) ignorent
`user_preferences.language`.

**Inclusivité française** — appliquée correctement (humain·e·s
seulement) : `utilisateur·ice·s`, `auteur·rice`, `lecteur·ice`. Pas
d'accident type `livre·s` ou `code·s` repéré. À conserver en
règle dans la suite.

## Tier 1 — combler les namespaces vides (cheap wins)

**Objectif** : passer de 7/13 à 13/13 namespaces remplis. Ces
chaînes existent déjà en français dans le JSX, il suffit de les
extraire et de les pointer.

**Pré-requis** : aucun. Le provider gère déjà tout.

**Coût** : 1–2 jours de travail mécanique, 6 PRs séparées.

### 1.1 — `account` namespace

Toutes les feuilles de
[`app/flow/Account/`](../../packages/web/src/app/flow/Account/).
~25 chaînes : titres de tabs, labels de DescribedSection (ils
contiennent déjà la copie complète, juste à externaliser),
messages de feedback (« Identifiant mis à jour », « Format
invalide », « Erreur lors de la modification »), CTAs.

- [ ] Créer `account.json` avec arborescence
      `{ tabs, sections, feedback, actions }`.
- [ ] Externaliser les 6 titres de tab (Identité, Sécurité, …) +
      `topbar.title`.
- [ ] Externaliser les 5 `DescribedSection` de
      [`SecurityTab.tsx`](../../packages/web/src/app/flow/Account/views/SecurityTab.tsx)
      (Code récup, Mot de passe, TOTP, Passkey, Mode sécurité) —
      `description` + `cta` séparés.
- [ ] Idem pour les 5 autres tabs (Identity, Recovery, Modules,
      Sessions, Danger).
- [ ] Tester en bascule FR↔EN : aucun fallback `defaultValue`
      visible côté EN.

### 1.2 — `common` namespace

Boutons et états réutilisés partout (Enregistrer, Annuler,
Supprimer, Modifier, Ajouter, Charger, Réessayer, …). Une fois
en place, devient la première chose à check avant d'écrire un
nouveau bouton.

- [ ] Créer `common.json` avec `{ actions, states, time, units }`.
- [ ] Lister les 15–20 verbes/actions récurrents avec un grep
      `>(Enregistrer|Annuler|Supprimer|…)<`.
- [ ] Migrer les 3–4 hotspots les plus visibles
      (`ComposerModal`, `Account/*`, `Goals/composer`).
- [ ] Documenter dans `documentation/Internationalisation.md`
      le pattern « toujours regarder `common.actions.*` avant
      d'écrire un bouton ».

### 1.3 — `goals`, `mood`, `passage`, `errors`

Mêmes mécaniques que `account`. À traiter chacun en PR isolée,
dans l'ordre d'effort croissant.

- [ ] `goals.json` — Composer, dialogs (CarryOver, ConfirmDelete),
      labels de statut, pluriels (cf. Tier 2).
- [ ] `mood.json` — Composer, questions, mois, hints. Vérifier si
      [`packages/web/src/i18n/fr/Mood/questions.json`](../../packages/web/src/i18n/fr/Mood/questions.json)
      doit être fusionné dans `mood.json` ou conservé tel quel
      (probablement à fusionner — un seul namespace par module).
- [ ] `passage.json` — Journal entry form, prompts, list labels.
      Reuse possible avec Goals (mêmes patterns CRUD).
- [ ] `errors.json` — Centraliser les messages des codes machine
      (`invalid_token`, `account_not_activated`, etc.) que chaque
      page traduit aujourd'hui en dur. Voir Tier 4 pour la table
      contrat API↔front.

## Tier 2 — pluralisation propre

**Objectif** : remplacer les ternaires `count === 1 ? 'X' : 'Y'`
par un mécanisme qui marche aussi pour des langues avec plus de
2 formes (russe = 4, arabe = 6) et qui surface les variantes à
traduire dans les JSON.

**Pré-requis** : Tier 1 partiel (au moins `goals`, `journal`,
`mood`, `library` qui contiennent les 80 % des cas).

**Coût** : 2–3 jours. Helper à écrire + sweep des call-sites.

### Approche

Étendre le helper `t()` avec un cousin `tn(key, count, values?)`
qui choisit la sous-clé via `Intl.PluralRules` :

```json
// goals.json
{
  "list.count": {
    "one": "{count} objectif",
    "other": "{count} objectifs"
  }
}
```

```tsx
// usage
tn('goals.list.count', total, { count: total })
// → en FR, count=1 → "1 objectif" ; count=3 → "3 objectifs"
// → en EN, count=1 → "1 goal" ; count=3 → "3 goals"
// → en RU, count=22 → forme "few" automatiquement
```

### Tâches

- [ ] Étendre [`I18nProvider.jsx`](../../packages/web/src/i18n/I18nProvider.jsx)
      avec `tn(key, count, values?)`. La résolution choisit
      `key.<rule>` où `<rule>` ∈ `{zero, one, two, few, many, other}`
      selon `Intl.PluralRules(language).select(count)`. Fallback
      sur `key.other` si la sous-clé absente.
- [ ] Ajouter un test Vitest sur la résolution avec FR (one/other)
      + EN (one/other) + un cas avec `zero` explicite.
- [ ] Migrer les 5 hotspots les plus lourds
      (`Goals/index.tsx`, `Mood/index.tsx`, `Journal/index.tsx`,
      `Library/index.tsx`, `Habits/HabitCard`).
- [ ] Migrer les conjugaisons verbales
      (`affected.length === 1 ? 'va' : 'vont'` →
      `tn('goals.carryover.verb', count)` avec `one: 'va', other: 'vont'`).
- [ ] Pour le cas double-`s` de
      [`SecurityTab.tsx:12`](../../packages/web/src/app/flow/Account/views/SecurityTab.tsx#L12),
      passer la phrase entière à `tn` plutôt que de plurializer
      noun + adjectif séparément (sinon double interpolation
      fragile).

### Décision à figer

ICU MessageFormat (via `intl-messageformat` ou similaire) ferait
le travail nativement, avec syntaxe `{count, plural, one {…} other {…}}`,
mais alourdit le bundle (~10 kB gzip). On reste sur du `tn`
maison tant qu'on n'a pas de cas trop complexe (genre + nombre
combinés). À réévaluer si on dépasse FR + EN + une 3ᵉ langue à
règles riches (russe, polonais).

## Tier 3 — formats date / nombre / liste

**Objectif** : aucune chaîne `"Aujourd'hui"`, `"Hier"`, `"Janvier"`
ne reste dans le code. Les formats date passent par
`Intl.DateTimeFormat(language, …)` au lieu de
`Intl.DateTimeFormat('fr-FR', …)`.

**Pré-requis** : Tier 1 (les libellés relatifs vont dans `common.time`).

**Coût** : 1–2 jours.

### Tâches

- [ ] Renommer
      [`packages/web/src/core/i18n/date-fr.ts`](../../packages/web/src/core/i18n/date-fr.ts)
      en `date-format.ts`. Toutes les fonctions prennent désormais
      `language` en paramètre (ou lisent depuis le store).
      Remplacer les `'fr-FR'` hardcodés par la langue active.
- [ ] Externaliser `"Aujourd'hui"` / `"Hier"` dans `common.time` ;
      le formatter construit sa sortie via `t(...)` au lieu de
      littéraux.
- [ ] Supprimer
      [`packages/web/src/app/flow/Goals/lib/date-format.ts`](../../packages/web/src/app/flow/Goals/lib/date-format.ts)
      (la const `FRENCH_MONTHS`) et utiliser
      `new Intl.DateTimeFormat(language, { month: 'short' })` à
      la place.
- [ ] Supprimer le tableau de mois en dur dans
      [`ComposerModal.tsx`](../../packages/web/src/ui/dirk/ComposerModal.tsx)
      et utiliser le même helper. Test Vitest qui vérifie que
      `genMonthOptions('en').includes('January')`.
- [ ] Remplacer les `Number.toLocaleString('fr-FR')` épars (admin
      + Journal/SideColumn) par un helper `formatNumber(n,
      language)` ou inliner `Intl.NumberFormat(language)`.
- [ ] Ajouter un helper `formatList(items, language)` basé sur
      `Intl.ListFormat` pour les énumérations actuelles
      `items.join(', ')` (dispatcher Library, listing tags…).

### Décision à figer

`Intl.RelativeTimeFormat` (« il y a 2 jours », « dans 3 minutes »)
n'est utilisé nulle part. À introduire seulement quand un nouveau
besoin tombe — pas de migration préventive.

## Tier 4 — contrat erreurs API↔front

**Objectif** : une seule table de traduction des codes machine
API. Côté front, n'importe quelle page qui reçoit un code
inattendu affiche un message générique cohérent au lieu d'une
chaîne hardcodée par feuille.

**Pré-requis** : Tier 1 (`errors` namespace prêt).

**Coût** : 1 jour.

### Tâches

- [ ] Recenser exhaustivement les codes renvoyés par
      [`packages/api/src/routes/`](../../packages/api/src/routes/)
      (auth, modules-config, items, etc.). Établir la liste
      canonique dans
      [`packages/shared/src/error-codes.ts`](../../packages/shared/src/error-codes.ts)
      (à créer) — type `ApiErrorCode = 'invalid_token' | …`.
- [ ] Remplir `errors.json` (FR + EN) avec une clé par code.
      Conserver une clé `unknown` pour le fallback générique.
- [ ] Ajouter un helper `apiErrorMessage(code, t)` dans
      [`packages/web/src/core/api/`](../../packages/web/src/core/api/).
      Tous les `catch` côté pages remplacent leur switch local
      par cet helper.
- [ ] Migrer les 6 messages d'erreur de
      [`Login.tsx`](../../packages/web/src/app/pages/Login.tsx)
      en premier (cas le plus visible). Étendre ensuite à
      `Register`, `ChangePassword`, `Recover`, `Account/*`.
- [ ] Test Vitest qui vérifie qu'à toute valeur de
      `ApiErrorCode` correspond une clé dans les deux langues
      (failsafe contre l'oubli au moment d'ajouter un code).

## Tier 5 — emails (api package)

**Objectif** : les emails respectent
`user_preferences.language`. Un user en `en` reçoit un email de
reset password en anglais.

**Pré-requis** : Tier 4 (le pattern est le même — table de
traduction côté shared, lookup côté api).

**Coût** : 3–5 jours. Plus lourd parce que l'API n'a pas
d'i18n actuellement, et qu'on touche à 8 templates.

### Tâches

- [ ] Décider du mécanisme : table JSON
      `packages/api/src/services/email/i18n/<lang>.json` lue par
      un helper `emailT(language, key, values)` minimal (pas de
      provider React côté node, juste une fonction). Pas de lib
      externe — on a déjà tout ce qu'il faut.
- [ ] Migrer le template le plus utilisé
      ([`register-activate`](../../packages/api/src/services/email/templates/))
      en premier comme prototype, vérifier que le mécanisme
      tient la route.
- [ ] Au moment de l'envoi
      ([`packages/api/src/auth/mailer.ts`](../../packages/api/src/auth/mailer.ts)),
      lookup la `language` depuis `user_preferences` (chiffré —
      donc on peut **pas** le lire côté API sans la clé). **Cas
      bloquant** : la langue de préférence est dans le blob
      chiffré, le serveur ne peut pas la lire pour adapter
      l'email. Deux options :
  - (a) Stocker une copie en clair dans `users.email_language`
        (FR / EN, pas un secret), maintenue par l'API au moment
        où le client appelle `setLanguage`. Coût : une migration
        + un endpoint, simple. **Choix recommandé**.
  - (b) Détecter via `Accept-Language` au moment de la requête
        qui déclenche l'email (register, reset password). Marche
        mal pour les emails non-déclenchés-par-le-user (admin
        envoie un invite à son tour).
- [ ] Implémenter (a) : migration `users.email_language` + API
      `PATCH /me/preferences` qui sync les deux côtés (chiffré +
      colonne en clair).
- [ ] Migrer les 8 templates dans le nouveau mécanisme.
- [ ] Test e2e : créer un user EN, déclencher un reset password,
      vérifier que l'email reçu (Mailpit en dev) contient le
      template EN.

## Tier 6 — outillage qualité

**Objectif** : empêcher la régression. Tout nouveau hardcode
français dans le JSX est rejeté en review.

**Pré-requis** : Tier 1 + 4 livrés (sinon trop de faux positifs).

**Coût** : 1 jour.

### Tâches

- [ ] Ajouter un test Vitest qui charge les deux JSON de chaque
      namespace et vérifie qu'ils ont **exactement les mêmes
      clés** (un trou en EN = échec CI). Lever le `defaultValue`
      sur les call-sites au passage — il devient inutile et
      masquait des trous.
- [ ] Règle ESLint custom (ou `eslint-plugin-react-i18n`)
      contre les chaînes littérales > 3 caractères dans le JSX,
      sauf whitelist (icônes, séparateurs `·`, ponctuation).
      Tolère les exceptions via commentaire `// i18n-ignore`.
- [ ] Scripte un `pnpm i18n:diff` qui compare FR vs EN et
      affiche les clés présentes seulement d'un côté. Utile en
      review.
- [ ] Mettre à jour
      [`documentation/Internationalisation.md`](../../documentation/Internationalisation.md)
      avec la convention finale : structure namespace, helper
      `tn`, règle ESLint, contrat API↔front.

## Sequencing recommandé

1. **Tier 1.1 + 1.2** (account + common) en priorité — ce sont les
   fondations sur lesquelles le reste s'appuie, et les surfaces
   les plus visibles d'un user.
2. **Tier 2 (pluralisation)** dès que `goals` et `mood` sont en
   place — mieux vaut le faire avant de migrer les 5 modules
   restants pour éviter de re-passer sur les ternaires.
3. **Tier 1.3 (goals/mood/passage/errors)** + **Tier 4 (contrat
   erreurs)** en parallèle — les deux se nourrissent.
4. **Tier 3 (dates/nombres)** — peut se faire indépendamment dès
   que `common` existe, sans bloquer les autres.
5. **Tier 5 (emails)** — chantier à part, à lancer quand le reste
   est stable. Demande la décision sur `email_language` (option
   a vs b ci-dessus) avant tout code.
6. **Tier 6 (outillage)** — clôture. Ne pas l'ajouter avant que
   le code soit i18n-clean, sinon CI rouge sur tout le repo.

## Décisions à figer (avant de commencer)

- [ ] Bibliothèque ICU MessageFormat ou helper `tn` maison ?
      (Recommandation : maison tant qu'on est à FR + EN ;
      réévaluer à la 3ᵉ langue.)
- [ ] `users.email_language` (clair) vs `Accept-Language`
      runtime ? (Recommandation : colonne en clair, plus
      robuste.)
- [ ] Migration de [`I18nProvider.jsx`](../../packages/web/src/i18n/I18nProvider.jsx)
      en `.tsx` strict pendant le chantier ? (Le projet impose le
      TS strict pour le nouveau code ; ce fichier est un legacy
      JSX. Recommandation : migration en même temps que Tier 2,
      où on touche au provider de toute façon.)
- [ ] Stratégie nouvelle langue : on ouvre EN en parallèle du FR
      (donc chaque PR Tier 1 livre les deux) ou on fait FR seul
      d'abord, EN en passe groupée à la fin ? (Recommandation :
      les deux en parallèle — c'est le seul moyen de faire
      tomber les `defaultValue` au passage et de ne pas accumuler
      de dette de retraduction.)
