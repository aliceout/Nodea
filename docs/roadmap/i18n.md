# i18n — audit + roadmap

> **Statut** : audit posé, ré-audité après le chantier
> `factoring-audit`. Provider maison en place, FR + EN
> déclarés ; **6 namespaces sur 13 sont vides** côté
> `packages/web/src/i18n/locales/` et l'app contient
> **~870 lignes accentuées dans 168 fichiers** (UI + diagrammes
> Docs + module Review) + emails 100 % FR. Deux **bugs i18n
> actifs** sont à fixer en Tier 0 avant tout effort cosmétique.
> Cette roadmap décrit comment passer de « bilingue sur le
> papier » à « bilingue pour de vrai ».
>
> **Mise à jour** : à chaque PR qui livre une étape, cocher la
> case correspondante. Si une décision technique change la
> stratégie (lib externe adoptée, provider remplacé, etc.), mettre
> à jour `documentation/Internationalisation.md` dans le **même
> commit** — pas en suivi.

Audit mené en parallèle sur `packages/web/src/` (UI) et
`packages/api/src/` (templates email). Chiffres ré-audités au
commit `4e45616` (post factoring-audit). Référence

## État des lieux

### Infrastructure existante

[`packages/web/src/i18n/I18nProvider.jsx`](../../packages/web/src/i18n/I18nProvider.jsx)
expose un hook `useI18n()` :

- `t('namespace.key', { defaultValue, values })` — interpolation
  `{token}`, fallback automatique vers la langue par défaut (`fr`).
- `setLanguage(code)` — persiste dans `localStorage`
  (`nodea:language`) **et** dans `user_preferences` chiffré
  (sync cross-device une fois loggué·e).
- `availableLanguages` — alimenté par la const `SUPPORTED_LANGUAGES`
  du provider.

Détection initiale : `localStorage` → `navigator.language` → `fr`.

### Patterns problématiques

**Pluralisation par ternaire — 16 sites recensés** :
13 `=== 1 ?` (Goals/index, Journal/index, Mood/index,
Library/index, Review, Homepage, Mood/ScoreDonut,
Journal/SideColumn, ComposerModal/Journal × 2,
Goals/CarryOverDialog × 3) + 3 `> 1 ?` (Account/SecurityTab:77
double-`s`, Habits/HabitCard:68, Habits/Heatmap:90,
Passkeys/ListView:46, Totp/ListView:88-89). Aucun helper
centralisé.

```tsx
// app/flow/Goals/index.tsx
total === 1 ? 'objectif' : 'objectifs'
// app/flow/Account/views/SecurityTab.tsx (double accord)
{count} passkey{count > 1 ? 's' : ''} enregistrée{count > 1 ? 's' : ''}
// app/flow/Goals/components/CarryOverDialog.tsx:127-129
affected.length === 1 ? 'va' : 'vont'  // verbe conjugué
```

Une migration vers `Intl.PluralRules` ou ICU MessageFormat est un
prérequis avant tout effort de traduction sérieux. Voir Tier 2.

**Date hardcoding FR — 13 sites avec `'fr-FR'` en dur** + 4
sources distinctes de littéraux :

- [`core/i18n/date-fr.ts`](../../packages/web/src/core/i18n/date-fr.ts)
  — 5 occurrences `Intl.DateTimeFormat('fr-FR', …)` +
  `"Aujourd'hui"` / `"Hier"` en littéraux.
- [`app/flow/Goals/lib/date-format.ts`](../../packages/web/src/app/flow/Goals/lib/date-format.ts)
  — tableau `FRENCH_MONTHS`.
- [`ui/dirk/ComposerModal/lib/constants.ts:27-38`](../../packages/web/src/ui/dirk/ComposerModal/lib/constants.ts#L27-L38)
  — second tableau de mois.
- [`app/flow/Mood/lib/constants.ts:121`](../../packages/web/src/app/flow/Mood/lib/constants.ts#L121)
  — troisième tableau de mois (oublié par le 1er audit).
- Sites `'fr-FR'` dispersés : `Mood/lib/heatmap.ts` (×3),
  `Review/views/List.tsx`, `Admin/InviteCode.tsx`,
  `Admin/AnnouncementsManager.tsx`, `Admin/SourcesPanel.tsx`,
  `Journal/SideColumn.tsx`, `Passkeys/ListView.tsx`,
  `BypassConfirm.tsx`. Un seul site fait le bon choix :
  [`Homepage/context.tsx:145`](../../packages/web/src/app/flow/Homepage/context.tsx#L145)
  (`language === 'en' ? 'en-US' : 'fr-FR'`).
- Bonus : `"Aujourd'hui"` en dur hors helper dans
  [`ui/dirk/sidebar/SidebarNav.tsx:32`](../../packages/web/src/ui/dirk/sidebar/SidebarNav.tsx#L32).

**Codes machine traduits inline** — l'API renvoie des codes type
`'invalid_token'` / `'account_not_activated'` ; chaque page front
les traduit avec sa propre table en dur
([`Login.tsx:64-72`](../../packages/web/src/app/pages/Login.tsx#L64-L72)).
Le pattern se duplique dans Register, Recover, ChangePassword,
RequestReset. Aucune table partagée — voir Tier 4.

**Emails 100 % FR — 7 templates** (`invite`, `mfa-bypass`,
`password-reset`, `recovery-applied`, `register-activate`,
`security-mode-downgraded`, + le wrapper `layout.ts`). Aucune
infrastructure i18n côté API ; un seul `// future i18n pass`
dans [`email/templates/layout.ts:20`](../../packages/api/src/services/email/templates/layout.ts#L20).
Les emails déclenchés par le user (activation, reset password,
MFA bypass, invite) ignorent `user_preferences.language`.

**Inclusivité française** — appliquée correctement (humain·e·s
seulement) : `utilisateur·ice·s`, `auteur·rice`, `lecteur·ice`. Pas
d'accident type `livre·s` ou `code·s` repéré. À conserver en
règle dans la suite.

## Tier 2 — pluralisation propre — livré

**Objectif** : remplacer les ternaires `count === 1 ? 'X' : 'Y'`
par un mécanisme qui marche aussi pour des langues avec plus de
2 formes (russe = 4, arabe = 6) et qui surface les variantes à
traduire dans les JSON.

**Pré-requis** : Tier 1 partiel (au moins `goals`, `journal`,
`mood`, `library` qui contiennent les 80 % des cas).

**Coût** : 2–3 jours. Helper à écrire + sweep des call-sites.

### Approche

Étendre le helper `t()` avec un cousin `tn(key, count, values?)`
qui choisit la sous-clé via `Intl.PluralRules` :

```json
// goals.json
{
  "list.count": {
    "one": "{count} objectif",
    "other": "{count} objectifs"
  }
}
```

```tsx
// usage
tn('goals.list.count', total, { count: total })
// → en FR, count=1 → "1 objectif" ; count=3 → "3 objectifs"
// → en EN, count=1 → "1 goal" ; count=3 → "3 goals"
// → en RU, count=22 → forme "few" automatiquement
```

### Tâches — livré (commit `1c11e51`)

- [x] Helper `tn(key, count, options?)` ajouté dans
      [`I18nProvider.jsx`](../../packages/web/src/i18n/I18nProvider.jsx),
      logique pure dans
      [`translate.ts`](../../packages/web/src/i18n/translate.ts).
      Résolution : `<key>.<rule>` (Intl.PluralRules) → `<key>.other`
      → bare `<key>` string → `defaultValue`. `count` est auto-
      injecté dans `values.count`.
- [x] Test Vitest dans
      [`translate.test.ts`](../../packages/web/src/i18n/translate.test.ts)
      — 19 cas couvrant FR (one/other), EN (one/other), RU
      (one/few/many/other via Intl.PluralRules), fallbacks à
      chaque niveau, override de `values.count`.
- [x] 5 namespaces migrés vers la structure nested
      `xxx.{one,other}` : account.security.passkey.count,
      goals.topbar.label + goals.carryOver.summary,
      mood.topbar.label + mood.donut.ariaLabel + mood.donut.centerCount,
      passage.topbar.label + passage.side.streakDay,
      review.topbar.label.
- [x] 8 call-sites migrés du ternaire
      `count === 1 ? t(...one) : t(...other)` vers `tn(key, count)` :
      Account/SecurityTab, Goals/index + Goals/CarryOverDialog,
      Mood/index + Mood/ScoreDonut, Journal/index +
      Journal/SideColumn, Review/List.
- [x] **Cas double-`s` SecurityTab** résolu : la phrase entière
      `« {count} passkey enregistrée[s]. »` est maintenant une
      seule clé pluralisée (`account.security.passkey.count.one`
      / `.other`), plus deux interpolations imbriquées noun +
      adjectif.
- [ ] **Reliquat** : ~12 ternaires de plural vivent encore dans
      des surfaces non-i18n (Habits/HabitCard, Habits/Heatmap,
      Passkeys/ListView, Totp/ListView, Account/SecurityTab pour
      des cas non-passkey). Tomberont à `tn` quand ces namespaces
      passeront en i18n.

### Décision à figer

ICU MessageFormat (via `intl-messageformat` ou similaire) ferait
le travail nativement, avec syntaxe `{count, plural, one {…} other {…}}`,
mais alourdit le bundle (~10 kB gzip). On reste sur du `tn`
maison tant qu'on n'a pas de cas trop complexe (genre + nombre
combinés). À réévaluer si on dépasse FR + EN + une 3ᵉ langue à
règles riches (russe, polonais).

## Tier 3 — formats date / nombre / liste — livré

**Objectif** : aucune chaîne `"Aujourd'hui"`, `"Hier"`, `"Janvier"`
ne reste dans le code. Les formats date passent par
`Intl.DateTimeFormat(language, …)` au lieu de
`Intl.DateTimeFormat('fr-FR', …)`.

### Tâches — livré (commits `1c1e5a7` + `9173529`)

- [x] **`core/i18n/date-fr.ts` → `core/i18n/date-format.ts`**
      (commit `1c1e5a7`, `git mv`). Toutes les fonctions sont
      paramétrées par `language` :
      `formatEntryLabel(iso, today, { language, todayLabel, yesterdayLabel })`,
      `formatMonthLabel(yyyymm, language)`,
      `formatLongDate(rawIso, language)`. Plus un helper
      `intlLocale(language)` (fr → fr-FR, en → en-US) qui
      centralise le mapping BCP-47, et un `formatNumber(n, language)`
      construit sur `Intl.NumberFormat`.
- [x] **« Aujourd'hui » / « Hier » externalisés** dans
      `common.time.{today,yesterday}` (FR + EN). Le formatter prend
      les labels en argument plutôt que de les baker en dur ; les
      mappers Mood + Journal lisent les labels via `t()` dans le
      provider et les passent à `recordToEntry`.
- [x] **`Goals/lib/date-format.ts` FRENCH_MONTHS supprimé**
      (commit `9173529`). `formatDate(dateIso, language)` utilise
      `Intl.DateTimeFormat` directement. Test Vitest étendu avec
      EN (Jan/Aug/Dec).
- [x] **`ComposerModal/lib/constants.ts` MONTH_OPTIONS supprimé** —
      remplacé par une factory `genMonthOptions(language)` que le
      Goal body memoise sur `language`.
- [x] **`Mood/lib/constants.ts` 4 const FR-only supprimés**
      (DAY_NAMES_FR, SHORT_MONTHS_FR, MONTH_LABELS_LONG,
      MONTH_LABELS_SHORT). 2 helpers utilitaires ajoutés dans
      `core/i18n/date-format.ts` :
      `getMonthNames(language, 'long' | 'short')` (12-array) et
      `getDayNames(language, 'long' | 'short')` (7-array, rotated
      Mon → Sun). Mood/lib/stats.ts `computePatterns` prend
      désormais `language` comme 3ᵉ argument.
- [x] **`formatNumber(n, language)` ajouté** + sweep
      Journal/SideColumn (les ternaires `language === 'en' ? …`
      remplacés par `intlLocale(language)`). Mêmes correction dans
      Homepage/context.tsx.
- [ ] **Reliquat partiel** : 3 sites Admin (`InviteCode`,
      `AnnouncementsManager`) + `BypassConfirm.tsx` utilisent
      encore `Date.toLocaleString('fr-FR')` ; le `Review/views/List.tsx`
      `DRAFT_DATETIME_FMT` est aussi FR-locked. Demande un petit
      `formatDateTime(date, language)` (date+heure) que `date-format.ts`
      n'a pas encore. Tomberont à un Tier 3 part 3 ou en bricolage
      local.
- [ ] **`formatList(items, language)`** basé sur `Intl.ListFormat`
      — pas encore livré, à ajouter quand un call-site qui ferait
      `items.join(', ')` apparaît clairement.

### Décision figée

`Intl.RelativeTimeFormat` (« il y a 2 jours », « dans 3 minutes »)
n'est toujours utilisé nulle part. À introduire seulement quand un
nouveau besoin tombe — pas de migration préventive.

## Tier 5 — emails (api package)

**Objectif** : les emails respectent
`user_preferences.language`. Un user en `en` reçoit un email de
reset password en anglais.

**Pré-requis** : Tier 4 (le pattern est le même — table de
traduction côté shared, lookup côté api).

**Coût** : 3–5 jours. Plus lourd parce que l'API n'a pas
d'i18n actuellement, et qu'on touche à 7 templates : `invite`,
`mfa-bypass`, `password-reset`, `recovery-applied`,
`register-activate`, `security-mode-downgraded`, + le wrapper
`layout.ts`.
