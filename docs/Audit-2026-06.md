# Audit complet — juin 2026

Audit du code applicatif (web + api) sur trois axes : **sécurité**, **performance**, **correctifs**.
Périmètre : exclusivement le code de l'application. Tout ce qui relève du serveur/VPS (nginx, en-têtes HTTP, docker, monitoring) est exclu — déjà couvert par les issues labellisées `server-side`.
Dédoublonnage fait contre les issues ouvertes (#59, #127, #128 notamment) : rien ici ne répète ce qui y est déjà tracé.

Chaque constat a été vérifié en lisant le code réel (fichier + ligne cités). Classement par gravité réelle pour l'utilisateur·ice, pas par catégorie technique.

## 🔴 Priorité 1 — à corriger en premier

### 1.1 Déconnexion : la session côté serveur peut ne jamais être invalidée
`packages/web/src/ui/layout/Layout.tsx:74-77`
Quand on se déconnecte depuis l'écran « clé manquante », la page navigue vers `/login` **avant** que la requête de déconnexion ait abouti — le navigateur peut l'annuler en plein vol. Résultat : le cookie de session reste valide côté serveur. Sur un ordinateur partagé, la personne suivante peut rouvrir l'app authentifiée.
**Correctif** : supprimer la redirection manuelle et attendre la fin de `session.logout()` (qui redirige déjà tout seul). Une ligne.

### 1.2 Sentry : les filtres de confidentialité existent mais ne sont pas branchés (web ET api)
`packages/web/src/sentry.ts:70-93` · `packages/api/src/sentry.ts:80-108`
Des filtres complets et testés existent (`sentry-scrub.ts` des deux côtés) mais **ne sont jamais importés**. Si un opérateur configure Sentry, partent vers ce service tiers : le nom du module actif (Mood, Journal…), les URLs d'appels réseau, et côté api **le texte des recherches de livres en clair** (dans les URLs sortantes vers Amazon/Google). C'est exactement ce que l'architecture s'efforce de cacher partout ailleurs.
**Correctif** : appeler `scrubSentryEvent` dans les deux `beforeSend`, désactiver les miettes de navigation (« breadcrumbs ») HTTP, corriger les commentaires périmés, ajouter un test.

### 1.3 API : aucune limite de taille sur les requêtes → une seule requête peut faire tomber le serveur
`packages/api/src/app.ts` (aucun `bodyLimit` monté) · `packages/api/src/routes/records.ts:396-422`
Les limites Zod (8 Mo par enregistrement, 16 Mo par lot) ne s'appliquent **qu'après** que le corps complet est chargé en mémoire. Sur `/records/bulk`, la vérification du total se fait après la validation de chaque élément : 100 éléments de 8 Mo (~800 Mo) sont matérialisés en mémoire avant le rejet. Une personne authentifiée peut faire planter l'api avec une seule requête.
**Correctif** : monter le middleware `hono/body-limit` globalement (~20 Mo) avant les routes.

### 1.4 Éditeur visuel : le brouillon restauré s'affiche vide et peut être perdu silencieusement
`packages/web/src/ui/dirk/forms/MarkdownEditor.tsx:100-110`
L'éditeur visuel ne se remplit qu'au montage. Quand un brouillon Journal/Goals est restauré (déchiffrement asynchrone, donc après le montage), **l'éditeur reste vide à l'écran alors que le brouillon est dans l'état**. Si on tape un caractère, le brouillon est écrasé ; si on enregistre, on sauvegarde un contenu qu'on n'a jamais vu. Le bouton « Repartir à zéro » a le bug inverse : le texte reste affiché alors que l'état est vide.
**Correctif** : re-synchroniser l'éditeur quand la valeur change de l'extérieur (sauf pendant que l'utilisateur·ice tape), ou remonter l'éditeur via une `key` à la restauration/reset.

### 1.5 Changement d'email : la personne ne peut plus jamais se connecter
`packages/api/src/routes/auth-account.ts:267-290`
L'enveloppe d'authentification (OPAQUE) est liée à l'email d'inscription. La route de changement d'email met à jour l'email **sans re-créer l'enveloppe** : toute connexion ultérieure échoue avec « mot de passe invalide ». Le commentaire du code reconnaît le manque, mais la route est active. En bonus : aucun email de notification à l'ancienne adresse, aucune révocation des autres sessions.
**Correctif** : désactiver la route tant que la re-création OPAQUE n'est pas câblée, ou la câbler. Ajouter notification + révocation comme pour le changement de mot de passe.

### 1.6 Réinitialisation de mot de passe : les anciens liens restent utilisables
`packages/api/src/auth/reset-tokens.ts:13-24` · `packages/api/src/routes/auth-reset.ts:239-277`
Deux demandes de réinitialisation créent deux liens valides en parallèle ; après un reset réussi, l'autre lien reste pleinement utilisable pendant sa fenêtre d'une heure, et le lien consommé peut même resservir (le marquage « utilisé » tombe sur un jeton arbitraire).
**Correctif** : à la réussite d'un reset, invalider **tous** les jetons de reset de l'utilisateur·ice dans la même transaction.

## 🟠 Priorité 2 — bugs réels à fort impact

### 2.1 Préférences chiffrées : deux courses qui perdent des données
`packages/web/src/core/api/preferences-client.ts:40-42` · `use-preferences.ts:44-76` · `AnnouncementsCard.tsx:53-59`
- Un blob de préférences indéchiffrable est traité comme « pas de préférences » → la première écriture suivante **écrase tout** (thème, langue, annonces masquées) côté serveur.
- Le blob est lu-modifié-écrit en entier : masquer une annonce pendant qu'une lecture est en vol (ou depuis deux onglets) fait réapparaître l'annonce.
**Correctif** : distinguer « blob absent » de « déchiffrement échoué » (lecture seule dans ce cas) ; fusionner `dismissedAnnouncements` (union) au lieu d'écraser.

### 2.2 Bibliothèque : remplacer la couverture d'un livre existant ne fait rien
`packages/web/src/app/flow/Library/components/item-form/save.ts:161`
Le mode « Couverture seule » est proposé à l'édition, l'aperçu se met à jour, mais à l'enregistrement la nouvelle couverture est silencieusement jetée dès que le livre en a déjà une. UI trompeuse.
**Correctif** : câbler le remplacement (supprimer l'ancien blob, créer le nouveau, re-pointer), ou masquer le mode quand une couverture existe déjà.

### 2.3 Journal : « Repartir à zéro » ne purge pas les images, et le brouillon ressuscite
`packages/web/src/app/flow/Journal/components/JournalForm.tsx:256-261`
Le reset vide le fil et le texte mais pas les pièces jointes ; l'auto-sauvegarde se redéclenche 800 ms plus tard avec les images restantes et **ré-écrit le brouillon qu'on vient de supprimer**.
**Correctif** : ajouter `setAttachments([])` au reset.

### 2.4 Vider un module : quatre lacunes
`packages/web/src/core/modules/wipe-module.ts` · `ModulesManager.tsx:121-307`
- Les brouillons locaux ne sont pas purgés : après avoir « tout vidé », le formulaire restaure un brouillon issu des données effacées.
- Les compteurs de version ne sont pas rafraîchis (l'écran final demande de « recharger le module » à la main).
- Les sous-composants `WipePanel`/`Toggle` sont définis **dans** le corps du parent : chaque re-render du parent les démonte (mot de passe effacé, écran de progression perdu).
- En cas d'échec à mi-parcours, on ne sait pas quelles collections ont été vidées.
**Correctif** : purger les brouillons du module vidé, bumper les versions, extraire les sous-composants au niveau module, surfacer le détail par collection en cas d'échec.

### 2.5 Brouillons : jamais purgés à la déconnexion
`useJournalDraft.ts:29` · `useGoalDraft.ts:25` · `Review/hooks/useDraft.ts:21` · `use-session.ts:119-138`
Le contenu est bien chiffré (vérifié), mais les noms des clés de stockage et leurs horodatages révèlent « cette personne utilise Journal/Goals/Review, dernière activité à telle date » à quiconque ouvre le navigateur après déconnexion. Le projet a déjà corrigé exactement cette classe de fuite pour le mode d'affichage Library. En bonus : la clé de stockage n'est pas séparée par compte (deux comptes sur le même navigateur s'écrasent mutuellement leurs brouillons).
**Correctif** : à la déconnexion (et à la suppression de compte), supprimer toutes les clés `nodea:*:draft*` ; à terme, suffixer la clé par un dérivé non-réversible de l'utilisateur·ice.

### 2.6 Annulations de mutations : Library et Journal restaurent un instantané périmé
`use-library-actions.ts:138-212` · `Journal/context.tsx:352-359`
Mood et Goals annulent finement (par entrée) quand une requête échoue ; Library et Journal restaurent **la liste entière capturée avant** — annulant au passage toute autre action faite entre-temps (un favori basculé, etc.). Cas aggravé : si la suppression d'un livre échoue après celle de ses notes, des notes qui n'existent plus en base réapparaissent à l'écran.
**Correctif** : appliquer le même mécanisme d'annulation ciblée que Mood/Goals.

### 2.7 HRT : la création d'occurrences n'est toujours pas protégée entre deux onglets
`packages/web/src/app/flow/HRT/hooks/use-schedule-materialization.ts:66-124`
Le correctif récent est solide dans un onglet, mais deux onglets (ou deux appareils) ouverts en même temps peuvent encore créer des doublons — le serveur ne peut pas dédupliquer des blobs chiffrés. L'existence de l'outil de dédoublonnage confirme que le cas arrive.
**Correctif** : verrou inter-onglets (`navigator.locks`, déjà disponible dans tous les navigateurs cibles) autour de la passe de matérialisation.

### 2.8 Logs serveur : l'URL de couverture identifie le livre ajouté
`packages/api/src/middleware/sanitize-log-url.ts:59-62`
Le commentaire du fichier affirme que `/library/` est couvert par la censure des logs — c'est faux, le préfixe n'est pas dans la liste. À chaque ajout de livre avec couverture, l'URL du fournisseur (qui permet de retrouver le titre exact) part en clair dans les logs du serveur, horodatée. Divergence doc-vs-code au sens du CLAUDE.md.
**Correctif** : ajouter `'/library/'` aux préfixes censurés + un cas de test dans `log-opacity.test.ts`. Voir aussi : le message d'erreur Amazon embarque l'URL finale (qui peut contenir la recherche) — tronquer au chemin (`amazon.ts:123-127`).

### 2.9 Config des modules : déchiffrée sans validation
`packages/web/src/core/api/modules-config-client.ts:35-37`
Le blob le plus sensible du runtime (il porte les identifiants qui scopent toutes les requêtes) est parsé sans schéma, contrairement aux préférences qui sont validées par Zod.
**Correctif** : créer un `ModulesRuntimeSchema` dans `packages/shared` et valider au chargement.

## 🟡 Priorité 3 — performance (l'app à 500+ entrées)

### 3.1 Chaque mutation re-télécharge et re-déchiffre toute la collection
`versions.ts` + tous les contextes de modules
Étoiler un livre re-télécharge les 3 collections Library **couvertures comprises** (10-20 Mo à 300 livres). Cocher une habitude attend le re-déchiffrement de tous les logs. Créer une entrée Journal re-paye toutes les images. L'ironie : le code optimiste (mise à jour locale immédiate) existe déjà partout — il est suivi d'un refetch complet qui le rend inutile.
**Correctif** : ne plus déclencher le refetch après les mutations dont la réponse serveur fait foi ; insérer/remplacer localement l'enregistrement retourné. Garder le refetch pour les vraies resynchronisations.

### 3.2 Le formulaire Journal re-télécharge tout le journal (images comprises) pour des suggestions de fils
`JournalForm.tsx:169-195` · même motif dans `Goals/components/form/use-thread-tokens.ts:25-49`
À chaque ouverture du formulaire, un chargement complet de la collection… pour extraire la liste des fils, **qui existe déjà, calculée et memoïsée, dans le contexte parent** (`useJournalFilters().threads`). Dix lignes à supprimer, gain immédiat.

### 3.3 Memoïsation cassée dans Library et Goals : tout re-rend à chaque frappe
`Library/context.tsx:189,209` · `Goals/context.tsx:160,182`
Les `useMemo` dépendent d'objets recréés à chaque render — ils ne mémorisent donc rien. Chaque caractère tapé dans la recherche re-rend toutes les tuiles/lignes. Mood et Journal font correctement (dépendances champ par champ).
**Correctif** : aligner sur le pattern Mood/Journal. Mécanique, sans risque.

### 3.4 Édition d'un livre : toutes les couvertures re-téléchargées pour en afficher une
`item-form/use-existing-cover.ts:37-57`
Le contexte Library détient déjà la couverture exacte dans sa `Map` — la lire au lieu de re-lister toute la collection.

### 3.5 HRT : la même collection est chargée 2 à 3 fois au montage
`use-schedule-materialization.ts:66-69` · `SummaryView.tsx:45` · `HRT/index.tsx:54-69`
Et re-chargée à chaque bascule de sous-vue. **Correctif** : hisser le hook de données au niveau de la page et le partager.

### 3.6 Vues murales/grille/tableau de Library non virtualisées
`BookWall.tsx:24` · `BookGrid.tsx:40` · `BookTable.tsx:164`
À 300+ livres, des centaines d'images montées d'un coup. **Correctif** : virtualiser par rangées + `loading="lazy"` sur les images (gain immédiat à une ligne).

### 3.7 Recherche sans différé
Aucun `useDeferredValue` sur les champs de recherche — le filtre et le regroupement se recalculent dans le même rendu que la frappe. Trois lignes par module.

### 3.8 Pièces jointes Journal : le chantier de fond
Déjà tracé dans l'**issue #59** (migrer les pièces jointes vers une collection séparée, labellisée v3). L'audit confirme que c'est le facteur déterminant pour la tenue à 500+ entrées illustrées : chaque visite de la Home ou du Journal télécharge et déchiffre toutes les images. **Recommandation : remonter #59 de v3 à la prochaine milestone.**

### Internationalisation — l'app affiche du français aux utilisateur·ices anglophones
Le module **Library** entier et le module **HRT** n'ont pas de fichiers de traduction. Le formulaire Journal (pourtant dans un module 100 % traduit) a une douzaine de chaînes en dur (erreurs de validation, bannière brouillon, compteur d'images). La barre d'outils de l'éditeur partagé (« Gras », « Italique »…), les 3 amorces du formulaire Mood, le toast de report Goals (« X goals reportés ») et le message de fin de wipe court-circuitent tous `t()`. Détail complet dans le rendu de l'agent correctness.
**Correctif** : créer `library.json` + `hrt.json` (FR/EN), balayer les chaînes en dur des nouveaux formulaires.

### Accessibilité — régressions des nouveaux formulaires
- La plupart des champs des 5 formulaires inline n'ont **pas de nom accessible** (placeholder seul) : titre/fil Goals, les 3 positives Mood, tous les champs du formulaire livre, le champ Page des notes.
- Les erreurs s'affichent dans un bandeau global mais **aucun champ ne pointe dessus** (`aria-describedby`/`aria-invalid`) — la règle du projet l'exige et HRT montre le bon pattern.
- À la fermeture d'un formulaire, le focus tombe sur `<body>` au lieu de revenir au bouton déclencheur.
- L'autocomplete des fils n'expose pas l'option surlignée aux lecteurs d'écran (`aria-activedescendant`).

### Documentation ↔ code (règle fondamentale du projet)
- `CLAUDE.md` référence `docs/security-audit.md` **qui n'existe pas** — recréer ou re-pointer (ce fichier-ci peut reprendre le rôle).
- `sentry-scrub.ts` (web) prétend que le SDK n'est pas installé — faux.
- `sanitize-log-url.ts` prétend couvrir `/library/` — faux (cf. 2.8).
- L'en-tête de `guard-derivation.ts` décrit une purge au logout inexistante (cf. 4.10).

## 🔭 Long terme (chantiers à planifier, pas urgents)

- **Lier chaque blob chiffré à son contexte (AAD)** : aujourd'hui un serveur compromis peut échanger deux ciphertexts du même compte (deux entrées de journal entre elles, ou le blob de config contre celui des préférences) sans que le déchiffrement échoue. Le gap est documenté comme assumé dans le code, mais sans plan de fermeture. Concevoir un format v2 avec liaison `(user, collection, record)` et migration opportuniste à l'écriture.
- **Synchronisation incrémentale** : l'enveloppe `{data, meta}` est déjà réservée côté client — un jour, ne re-télécharger que ce qui a changé au lieu de tout re-lister.
- **Fraîcheur de re-auth stampée dès le login** : toute session volée dans les 5 minutes suivant une connexion passe les barrières destructives sans ressaisie du mot de passe. C'est le modèle documenté ; à reconsidérer (ne stamper qu'à la première preuve explicite).
